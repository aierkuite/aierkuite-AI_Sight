# AI 视觉对话助手网页应用计划

## Summary
在 `G:\qiniuyun` 根目录实现一个本地运行的网页应用：前端使用 Vite React，后端使用 FastAPI。用户在 Chrome/Edge 中打开页面后授权摄像头和麦克风，点击/按住说话，前端用浏览器语音 API 转文字，在用户提交时截取当前摄像头画面，后端调用 OpenAI-compatible 多模态接口生成回答，前端展示文字并用浏览器语音合成播报。

不做设计文档，不做 Mock 降级；缺少 API Key、模型名或接口异常时明确报错。

## Key Changes
- 在根目录创建 Vite React 前端，首屏就是可用应用界面：摄像头预览、说话按钮、识别文本、发送状态、回答区、最近对话列表、错误提示。
- 创建 FastAPI 后端，负责读取 `.env`、校验必填配置、保护 API Key、接收前端文本和截图、调用 OpenAI-compatible 多模态模型。
- 前端语音链路使用浏览器能力：`SpeechRecognition` 做中文语音识别，`speechSynthesis` 播报 AI 回复；目标浏览器为最新版 Chrome/Edge。
- 摄像头策略为“提问时截图”：默认抓取当前单帧；保留可抓取最近 1~3 帧的余地，以应对动态场景演示。截图在前端压缩后随问题发送，服务器不保存图片、语音或对话数据。
- 多轮上下文只保留最近 6 轮文本对话，并在每次请求中附带当前截图，兼顾追问能力和调用成本。
- AI 回答采用流式输出（SSE）：后端边生成边推送，前端边接收边展示，并尽早开始语音播报，降低首字延迟、提升交互流畅性。

## Interfaces
- 环境变量必填：`OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL`；`OPENAI_MODEL` 必须是支持图片输入的模型名。
- 后端接口：`POST /api/chat`
  - 请求包含：用户文本、当前截图 base64、最近 6 轮对话历史。
  - 响应采用 SSE 流式返回 AI 回复文本分片；异常时返回可展示的错误信息或状态。
- 后端启动时检查必填环境变量；缺失时启动失败并输出清晰中文错误。
- 前端不接触 API Key，不保存持久化历史；刷新页面后对话历史清空。

## Test Plan
- 启动检查：缺少任一必填环境变量时，FastAPI 给出明确错误；配置完整时前后端可正常启动。
- 浏览器权限：Chrome/Edge 能请求摄像头和麦克风权限，拒绝权限时页面显示可理解的错误提示。
- 核心流程：点击说话后能识别中文文本，提交时截取画面，后端流式返回回答，前端边收边展示并尽早播报。
- 验收场景：识别日常物品、读取纸面文字、解释屏幕或物体细节、针对上一轮回答继续追问。
- 流畅性验证：观察从提交到首字出现的延迟，确认流式输出与边收边播生效。
- 成本策略验证：每次提问只发送一张（必要时最多 3 张）压缩截图，只发送最近 6 轮历史，不落盘图片、语音或对话。

## Assumptions
- 运行方式仅面向本地开发和课程展示，不做公网部署。
- 浏览器 `SpeechRecognition` 依赖联网（Chrome 走云端识别），且需在 localhost 或 https 环境下运行；展示环境需保证网络可用。
- UI 和提示语默认使用简体中文；用户用其他语言提问时，模型可按用户语言回答。
